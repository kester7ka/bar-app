# Bar Manager

Минималистичное веб-приложение для бара: учёт позиций и контроль сроков годности,
с просмотром графика смен и расчётом примерного заработка.

С версии **0.3.0** приложение работает только с авторизацией: незарегистрированный
пользователь видит только экран регистрации/входа.

## Запуск

**Самый простой способ (Windows):**

- Двойной клик по [start.bat](start.bat) — поставит зависимости (один раз),
  создаст БД, зальёт список баров и поднимет сервер на `http://127.0.0.1:5000/`.

**Вручную:**
```bash
cd backend
pip install -r requirements.txt
python seed_bars.py    # один раз — загрузить список баров (необязательно: бары
                       # засеются автоматически при первом старте сервера)
python server.py       # запустить сервер
```

Открыть приложение можно по адресу `http://127.0.0.1:5000/` либо открыв `index.html`
напрямую — он по умолчанию ходит на `127.0.0.1:5000`.

Если бэкенд развёрнут на другом адресе, добавь в `<head>` строку
`<script>window.BAR_APP_API = 'https://my-server';</script>` перед подключением
`js/api.js`.

## Админ-аккаунт

Администратор создаётся автоматически при старте сервера, если заданы
переменные окружения:

```
BAR_APP_ADMIN_USER=admin
BAR_APP_ADMIN_PASSWORD=надёжный-пароль
```

Локально — в `backend/.env`, на Railway/хостинге — в Variables. После старта
зайди обычным **Войти** с этим логином/паролем. У админа в Профиле появляется
панель:
- **Активный бар** — переключение между барами; админ видит позиции и график
  любого выбранного бара (через заголовок `X-Bar-Id`, который сервер принимает
  только от админов).
- **Генерация ключей** — выбираешь бар, количество, опц. заметку → получаешь
  8-значные коды для раздачи сотрудникам.

## Регистрация по одноразовому ключу

1. Одноразовые 8-значные ключи генерирует **админ** (см. выше). Каждый привязан к бару.
2. Сотрудник на экране входа жмёт **Регистрация**, вводит ключ, никнейм, пароль и принимает условия.
3. После регистрации ключ сгорает — повторно использовать его нельзя.
4. На следующие входы используется **Войти** с никнеймом и паролем.

Изоляция:
- Позиции, открытие/закрытие, удаление — всё видно только пользователям того же бара.
- На экране **График** автоматически показывается только бар пользователя,
  селектор баров скрыт.

## Логика TOB

- Каждая позиция имеет **ровно 6 цифр (0–9)**, уникальные в пределах бара.
- Уникальность одновременно проверяется и в браузере, и в БД (`UNIQUE(bar_id, tob)`).

## Открытые / закрытые

- Открытой может быть только **одна** позиция с одинаковым названием и категорией
  в баре. Гарантируется частичным уникальным индексом
  `idx_positions_one_open(bar_id, lower(name), category) WHERE is_open = 1`.
- При попытке открыть вторую — сервер вернёт 409.

## График смен

- Источник — публичный xlsx на Яндекс.Диске: `https://disk.360.yandex.ru/d/YPIq80g1M7G1SA`.
- Парсится прямо в браузере через SheetJS (CDN). Кэш в localStorage — 30 минут.
- Старший автоматически определяется по колонке «Должность»: «Ст. …» / «Старш…» →
  ставка **440 ₽/ч**, иначе **420 ₽/ч**. Расчёт примерный, до вычета налогов.
- Если в `users.bar_id` есть бар — селектор баров на экране графика скрывается,
  показывается только этот бар.

## Безопасность

**Хранение паролей.** Хэшируются `pbkdf2_sha256` с уникальной солью и
200 000 итераций. Пароли никогда не возвращаются в API-ответах.

**Сессии.** Токены — `secrets.token_urlsafe(36)`, 60 дней TTL. Хранятся в
таблице `sessions`, при логауте удаляются.

**Авторизация.** Все API-эндпоинты, кроме `/api/auth/register`,
`/api/auth/login`, `/api/health` и `/api/bars`, требуют
`Authorization: Bearer <token>`. Middleware `@require_auth` валидирует
токен и подтягивает `user_id` + `bar_id` из сессии.

**Изоляция баров.** Каждая операция дополнительно фильтруется по
`bar_id` пользователя из сессии — клиент не может передать чужой
`bar_id`, даже если знает чужие id позиций. Сервер вернёт 404.

**Защита от перебора.** На `/api/auth/register` и `/api/auth/login` —
in-memory rate-limit: до 10 неудачных попыток с одного IP за 5 минут.
Превышение — 429. На один Flask-процесс хватает; для многопроцессного
деплоя замените на Redis.

**Timing-safe логин.** Если ника не существует, всё равно прогоняется
pbkdf2 с «болванкой», чтобы по времени ответа нельзя было определить,
есть ли такой ник в системе.

**Защита от DoS.** Лимиты длин на стороне сервера:
- username: 3–32 символа
- display_name: до 64
- password: 6–128
- name позиции: до 120

Без них клиент мог бы прислать гигантскую строку и нагрузить хеш.

**Строгие булевы поля.** `is_open` и `accepted_policy` принимаются только
как Python `True` (из JSON `true`). Строки `"false"`, `"0"` и пр. дают
`False`. Раньше `bool("false")` давал `True` — баг был.

**Security-заголовки** на каждый ответ:
- `X-Content-Type-Options: nosniff`
- `X-Frame-Options: DENY`
- `Referrer-Policy: same-origin`
- `Permissions-Policy: geolocation=(), microphone=()`

**CORS.** По умолчанию `*` для локальной разработки. Для продакшена
поставь `BAR_APP_CORS_ORIGINS=https://bar.example.com` в `.env` —
сервер начнёт пропускать только перечисленные origins.

**Что не уходит наружу.** SQL-ошибки больше не возвращаются в теле
ответа — пишутся в лог сервера, клиенту приходит общий `«Не удалось
сохранить позицию»`. Это убирает утечку структуры БД.

**Чего ЕЩЁ нет (стоит добавить перед продом):**
- HTTPS — Flask по умолчанию HTTP, разворачивай за nginx/Caddy с TLS
- Argon2 вместо PBKDF2 — современнее, но требует доп. пакета
- Redis-rate-limit вместо in-memory — для многих воркеров
- Двухфакторка / повторная аутентификация для чувствительных действий
- Логи в файл с ротацией (сейчас stdout)
- Подпись токенов (JWT) с проверкой целостности — у нас простой opaque
  токен в БД, что безопасно, но требует round-trip к БД на каждый запрос
